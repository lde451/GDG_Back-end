## 계층형 아키텍처
스프링은 애플리케이션을 역할에 따라 계층으로 나눠서 설계한다. 이를 계층형 아키텍처라고 하는데, 각 계층이 자기 역할만 담당하기 때문에 코드가 훨씬 깔끔해지고 유지보수하기 쉬워진다.

Controller는 클라이언트의 HTTP 요청을 가장 먼저 받아서 Service에 전달하고 결과를 응답으로 돌려주는 역할을 한다. 직접 비즈니스 로직을 처리하지 않고 요청을 전달하는 역할에만 집중한다. Service는 실제 비즈니스 로직을 처리하는 핵심 계층이다. 회원 정보 수정이나 상품 구매처럼 여러 단계를 거치는 작업들이 여기서 이루어진다. Repository는 DB에 직접 접근해서 데이터를 조회하거나 저장하는 역할을 하며, Service가 필요한 데이터를 요청하면 DB에서 꺼내다 준다.

## DTO vs Entity
DTO는 계층 간 데이터를 주고받을 때 사용하는 객체로, 소통 목적에 맞는 필요한 정보만 담는다. Entity는 DB 테이블과 매핑되는 객체로 모든 정보를 담고 있다. 보안과 데이터 일관성을 위해 Entity를 외부에 직접 노출하지 않고, DTO로 변환해서 전달하는 방식을 사용한다.

## 패키지 구조
패키지 구조를 잡는 방식은 크게 두 가지다. 계층형은 Controller, Service, Repository끼리 모아서 관리하는 방식이고, 도메인형은 Member, Product처럼 도메인 단위로 관련 클래스를 한 곳에 모아서 관리하는 방식이다. 도메인형은 새로운 기능을 추가할 때 다른 도메인에 영향을 덜 주고 코드를 찾기도 쉬워서 일반적으로 도메인형이 권장된다.

## 상태 코드 & ResponseEntity
HTTP 응답에는 요청 처리 결과를 나타내는 상태 코드가 포함된다. 200은 요청 성공, 201은 새로운 리소스 생성 성공, 204는 성공했지만 반환할 데이터가 없을 때(삭제 등), 400은 클라이언트의 잘못된 요청, 404는 리소스를 찾을 수 없을 때, 500은 서버 내부 오류를 의미한다. ResponseEntity를 사용하면 상태 코드와 응답 데이터를 함께 유연하게 반환할 수 있다. 예를 들어 생성 요청이면 created(uri).build()로 201 응답을, 삭제 요청이면 noContent().build()로 204 응답을 보낼 수 있다.

## 스프링 빈 & 의존성 주입
스프링 컨테이너는 빈을 생성하고 관리하는 공용 창고 같은 역할을 한다. 빈은 애플리케이션 전역에서 공유되는 객체로, 필요할 때마다 새로 만드는 게 아니라 이미 만들어진 객체를 재사용한다. @Component, @Controller, @Service, @Repository 어노테이션을 붙이면 스프링이 자동으로 해당 클래스를 빈으로 등록해준다.

의존성 주입은 필요한 객체를 직접 생성하지 않고 스프링 컨테이너가 대신 만들어서 주입해주는 방식이다. 주입 방법에는 생성자 주입, 필드 주입, 수정자 주입이 있는데 생성자 주입이 가장 권장된다. final 키워드로 불변성을 보장할 수 있고 테스트하기도 편하기 때문이다. Lombok의 @RequiredArgsConstructor를 사용하면 final 필드에 대한 생성자를 자동으로 만들어줘서 코드가 훨씬 간결해진다.

## 트랜잭션
여러 DB 작업이 하나의 논리적인 단위로 묶여야 할 때 사용한다. 예를 들어 주문 처리 중 결제, 재고 감소, 주문 내역 저장이 모두 이루어져야 하는데, 중간에 하나라도 실패하면 전체가 롤백되어야 한다. 이런 원자성을 보장해주는 게 트랜잭션이다. @Transactional 어노테이션으로 적용하며, 조회만 하는 경우 readOnly = true 옵션을 추가하면 성능을 최적화할 수 있다.
